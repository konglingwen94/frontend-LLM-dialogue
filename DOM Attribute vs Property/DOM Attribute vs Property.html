HTML 中的布尔属性（如 checked, disabled, readonly）是「存在即为真」，根本不关心你赋的值是什么：

vue
<input :checked="false" />

相当于：

js
el.checked = false

这样 input 就不会被选中，避免了字符串转布尔的歧义。

❗一致性抽象的典型副作用（以 Vue 为例）

1️⃣ Debug 难度提升：抽象越深，越难还原底层因果链

在 Vue 中，很多属性（如 v-model、:checked）都不是简单的 DOM 绑定，而是经过响应式系统、生命周期钩子、虚拟 DOM diff 等多层抽象机制处理的。

这导致：

  • 看起来是个简单的绑定，实际渲染路径非常长。

  • 出现 bug 时，不能“直觉式”地从 HTML 找问题，而要了解“整个响应链条”。

🧠 这对调试者提出了新要求：不仅要懂 DOM，还得懂 Vue 的实现机制。

2️⃣ 心智模型错位：框架“营造的世界”与浏览器“真实运行世界”不完全等价

举个例子：

vue
<input :value="msg" />

这在 Vue 中看起来像是“你控制了 input 的 value”，但实际：

  • DOM 元素的 .value 和 value attribute 是两码事；

  • 如果用户手动修改了 input 的内容，Vue 可能会在下次更新中**“还原”**它；

  • 如果你用 v-model.lazy，更新又变成“等到 blur 再同步”。

🧠 这时候，你以为你在控制 input，实际上是 Vue 在控制 input，而你在控制 Vue。
如果你不理解其中的时序与控制权分配，容易产生“失控”感。

3️⃣ 抽象叠加后可组合性降低

当多个抽象层（如组件 → 插件 → 指令 → 响应式系统）组合在一起时：

  • 某个 prop 的异常行为可能来自上层组件注入，也可能是指令修改了 DOM。

  • 多个一致性策略之间产生冲突时（如 prop 和 slot 同时提供数据），问题变得难以归因。

🧠 一致性带来了“封装感”，但封装过度则让内部机理变得模糊。

4️⃣ 强一致性牺牲灵活性：不允许你“绕过框架”

Vue（尤其是 3.x 的 Composition API）倾向于全面控制状态与视图：

  • 你直接操作 DOM 可能会失效（因为下一轮 patch 会还原）；

  • 你不按框架方式组织响应式数据，某些行为就“不响应”；

  • 某些 native 行为（如表单默认行为）会被 Vue 的控制逻辑覆盖。

🧠 一致性系统的前提是“你不要越界”，但这也意味着开发者的实验空间被压缩。

🧭 更深一层的反思：一致性不是目的，而是一种「范式工具」

一致性抽象本质上是一种“认知压缩策略”：让你用更简单的概念操作更复杂的行为。

但：

  • ❌ 它不能也不应屏蔽一切差异。

  • ✅ 真正理想的设计是允许开发者在需要时“下探”到抽象之下的原始行为层。

也就是说：

一致性是基础体验，但透明性和可控制性才是高级能力。

✅ 好的框架如何避免副作用？

以 Vue 为例，它做了一些平衡尝试：

特性提供一致性允许还原控制
v-model双向绑定、统一语法.lazy, .trim, .number 等修饰符提升语义可控性
组件 props接收外部状态defineExpose, emits, provide/inject 暴露内部钩子
模板语法声明式模板ref, watchEffect, onMounted 等回归命令式控制
自动响应式状态自动跟踪markRaw, shallowRef, toRaw 控制粒度细节

这些机制并非“违反一致性”，而是赋予开发者对一致性层级的选择权。

一致性带来的好处一致性引发的问题
